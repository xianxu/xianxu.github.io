### The core insight is right: colocate state with code

The fundamental problem you've identified is that **context is the scarcest resource in agentic coding**, and scattering it across systems (Linear, Jira, local ephemeral files, conversation history) means every agent session starts with a costly reconstruction phase. Making the repo the single source of truth for both code *and* work state is sound — it gets you versioning, diffing, branchability, and resumability for free.

### On your proposed structure

The structure is reasonable, but I'd push on a few things:

**The `tombed` marker feels like reinventing git.** If an issue is done, that's really a property of the commit history — the issue folder existed, work was done, and the resolution is the diff. A simpler model might be: issues that exist in `issues/` are open; closing an issue means deleting the folder (the history lives in git). Or if you want to keep them around for reference, a simple `status` file or frontmatter in `issue.md` works. The tombstone file approach creates a state you have to check that's redundant with what git already tracks.

**`todo.md` is doing a lot of work.** This is really the *plan* — the decomposition of the issue into steps. I'd be more explicit about its semantics. Is it a checklist the agent checks off? Is it append-only? Can the agent revise it mid-execution? The interesting design question is: **should the agent be allowed to modify the plan, or only execute it?** In practice, plans change as you learn things during implementation. You probably want the agent to be able to re-plan, but you want that to be an explicit, visible action (a diff you can review), not something that silently happens.

**Dependencies as issue IDs is good but incomplete.** You'll likely need to distinguish between:
- Hard dependencies (must be merged first, the code won't work otherwise)
- Soft dependencies (would be easier if done first, but can be worked around)
- Conflicts (touches the same files, can't be parallelized safely)

### The parallelism problem is the real hard problem

Your instinct toward single-threaded execution is pragmatic, but I think it's worth being more precise about *why* parallelism is hard here. It's not just merge conflicts — it's **semantic conflicts**. Two agents can modify different files but break each other's assumptions. 

The structured way to bring parallelism back might not be repo separation but rather **scope separation within a repo**:

- **By module/package boundary**: if your codebase has well-defined module boundaries, two issues touching different modules can safely parallelize. The agent needs to know the dependency graph of *modules*, not just issues.
- **By layer**: an API contract change can be split into "update the interface" (serial, blocking) and "update all callers" (parallelizable per caller).
- **Speculative execution**: run N agents in parallel on independent issues, but against the *same base commit*. Then merge sequentially, and if a later merge has conflicts, re-run that agent against the new base. This is essentially optimistic concurrency control.

The speculative execution model is probably the most practical — you accept that some work will be thrown away, but the wall-clock speedup is worth it for issues that *usually* don't conflict.

### What might be missing

One thing I don't see in your model: **the agent's reasoning trace**. When you come back to a half-finished issue, the todo.md tells you *what* was planned, but not *why* certain decisions were made. You might want something like `00001/log.md` that captures key decisions and discoveries during execution. This is essentially the agentic equivalent of commit messages, but at a higher level of abstraction.

Also worth considering: who *creates* the issue decomposition? If a human writes `issue.md` and the agent generates `todo.md`, that's a clear contract. If the agent does both, you need a review step before execution begins.

Here's what I'd propose — biased toward minimalism, with clear conventions over configuration.

### Structure

```
issues/
  0001/
    issue.md
    plan.md
    log.md
    deps
    status
```

That's it. Five files max per issue.

### File semantics

**`issue.md`** — The problem statement *and* what done looks like. One file. Keep them together because the definition of done is meaningless without the problem context. Format:

```markdown
# Auth tokens not refreshed on 401

When the API returns 401, the client crashes instead of refreshing.

## Done when

- Client intercepts 401, refreshes token, retries original request
- Test covers the refresh-and-retry path
- No other callers affected
```

**`plan.md`** — The current execution plan. A checklist. The agent owns this file and can revise it. Revisions are visible in git diff, which is your audit trail. No separate versioning needed.

```markdown
- [x] Identify all call sites that hit authenticated endpoints
- [x] Extract token refresh logic from login flow
- [ ] Add retry interceptor to HTTP client
- [ ] Write test for 401 → refresh → retry
- [ ] Verify no regression in existing auth tests
```

**`<a href="{% post_url 2026-03-28-further_on_the_ideal_of_logmd %}" class="branch-inline">log.md</a>`** — Append-only. The agent writes here when something unexpected happens: a failed approach, a discovery, a decision that deviates from the plan. This is what makes resumability work — the next session reads this to understand *why* the plan looks the way it does.

```markdown
## 2025-06-28 14:30
Tried wrapping fetch() directly but the interceptor pattern 
doesn't work because we use a custom transport layer. 
Need to hook in at the transport level instead.
Revising plan step 3.
```

**`deps`** — One issue ID per line. Empty file or absent means no dependencies.

```
0003
0007
```

**`status`** — A single word on a single line. Three possible values:

```
open
```

- **`open`** — ready to work on (or being worked on; in single-threaded mode, this distinction doesn't matter)
- **`blocked`** — can't proceed, and the reason isn't just deps (agent got stuck, needs human input, external blocker)
- **`done`** — finished

That's the entire lifecycle. The scheduler logic is trivial:

```
pick the first open issue whose deps are all done
```

### DAG decomposition

When a step in `plan.md` gets big enough that it could be committed, tested, and reviewed independently — pull it out into its own issue. The parent issue's plan then references the child:

```markdown
- [x] Extract token refresh logic → issue 0008
- [ ] Add retry interceptor → issue 0009
- [ ] Integration test
```

The parent's `deps` doesn't list its children (that would be circular). Instead, the parent stays `open` and its plan tracks children by reference. The parent is done when its plan is fully checked off.

Children *can* depend on each other via their own `deps` files. That's where your topological ordering comes from.

### Plan revision protocol

This is the part that matters more than file layout. Three rules:

1. **The agent can revise `plan.md` freely**, but must write an entry in `log.md` explaining why before making the change. This makes plan drift visible and reasoned.

2. **If the agent discovers the issue itself is wrong** (wrong problem statement, wrong scope), it sets status to `blocked` and writes in the log what needs human attention. It does not silently redefine the issue.

3. **If a plan step turns out to need more than ~2-3 commits**, the agent decomposes it into a sub-issue rather than inflating the current plan. This keeps each issue's plan short enough to hold in context.

### Committing strategy

Commit `issues/` to the repo, on whatever branch you're working on. When an issue is done and merged, you can either:

- Delete the folder in the merge commit (clean main, history in git)
- Leave it (simple, searchable, slightly noisy)

I'd start with deleting on merge. If you miss having the history around, switch to leaving them.

```bash
#!/bin/bash
# next-issue: prints the next issue to work on

for dir in issues/*/; do
  id=$(basename "$dir")
  status=$(cat "$dir/status" 2>/dev/null || echo "missing")
  
  if [ "$status" != "open" ]; then
    continue
  fi

  # Check all deps are done
  all_done=true
  if [ -f "$dir/deps" ]; then
    while IFS= read -r dep; do
      dep=$(echo "$dep" | tr -d '[:space:]')
      [ -z "$dep" ] && continue
      dep_status=$(cat "issues/$dep/status" 2>/dev/null || echo "missing")
      if [ "$dep_status" != "done" ]; then
        all_done=false
        break
      fi
    done < "$dir/deps"
  fi

  if [ "$all_done" = true ]; then
    echo "$id"
    exit 0
  fi
done

echo "No runnable issues found." >&2
exit 1
```

### What I deliberately left out

- **No YAML/JSON metadata.** Plain text files are easier for both humans and agents to read and write. You don't need structured data until you're building dashboards.
- **No separate acceptance criteria file.** It's three lines in `issue.md`. Another file is another thing to keep in sync.
- **No artifacts folder.** If the agent produces research or patches, they're either commits or they go in the log.
- **No priority field.** In single-threaded topological execution, priority is determined by dependency order and the sequence you create issues in. If you need explicit priority later, add a single number to the first line of `status` (e.g., `open 1`).

### Design: Issues as Markdown with YAML Frontmatter

Parley already has the pattern: chats are markdown files with a header block. Notes have templates and a directory structure. Issues should be the same thing — no new abstractions, just a new `issues_dir`.

### Directory layout

```
{issues_dir}/
  0001-auth-token-refresh.md
  0002-extract-parser.md
  0003-add-retry-tests.md
```

That's it. **One file per issue.** No subdirectories per issue. The plan, log, status, and deps all live inside the single markdown file as sections. This mirrors how a parley chat is one file with all state in it — the header plus the transcript.

### Issue file format

Following parley's convention of frontmatter + structured content:


### Why single file, not a folder-per-issue

Three reasons that align with parley's philosophy:

1. **Parley chats are single files.** A chat's entire state — header, system prompt reference, all turns — is one `.md`. Issues should follow the same pattern. You already have muscle memory for navigating a structured markdown file in nvim.

2. **Telescope picker works on files, not directories.** The existing [picker infrastructure](https://github.com/xianxu/parley.nvim/blob/main/specs/ui/pickers.md) can list issues the same way it lists chats or notes — `rg` or `fd` over a directory, show filename + maybe grep the status from frontmatter. No new picker abstractions needed.

3. **`@@` file references already work.** In a parley chat, you can reference `@@issues/0001-auth-token-refresh.md` to pull an issue into context when discussing it with an LLM. One file = one reference. If issues were folders, you'd need to reference multiple files or invent a new aggregation syntax.

### Scheduler logic (the `IssueNext` command)

```lua
-- Pseudocode for next runnable issue
function M.next_issue(issues_dir)
  local issues = parse_all_issues(issues_dir)
  local done_ids = set of issues where status == "done"

  for _, issue in ipairs(sorted_by_id(issues)) do
    if issue.status == "open" then
      local all_deps_done = true
      for _, dep in ipairs(issue.deps) do
        if not done_ids[dep] then
          all_deps_done = false
          break
        end
      end
      if all_deps_done then
        return issue
      end
    end
  end
  return nil
end
```

Parsing frontmatter is trivial — read the file, extract the YAML block between `---` markers, pull `status` and `deps`. Parley already parses chat headers, so this pattern exists in the codebase.

### The `IssueDecompose` command

This is the one non-obvious command. When your cursor is on a plan line like:

```markdown
- [ ] Extract token refresh logic from login flow
```

Running `:ParleyIssueDecompose` should:

1. Create a new issue file `0002-extract-token-refresh.md` (prompts for slug)
2. Copy the line as the issue title
3. Add the new ID to the parent's `deps` in frontmatter
4. Replace the plan line with: `- [ ] Extract token refresh logic → 0002`
5. Open the new issue for editing

This makes DAG decomposition a single keystroke from within the plan.

### What I deliberately kept aligned with parley's existing patterns

- **Frontmatter for metadata** — chats already use this
- **Emoji-free** — unlike chat turns (💬/🤖), issues don't need visual turn markers; markdown headings suffice
- **Picker-first navigation** — same as chat finder and note finder
- **`@@` compatible** — issues are just `.md` files, so existing file reference syntax works in chats
- **One file = one thing** — same as one chat file = one conversation

### What this does NOT include (deliberately)

- No priority field (ID order + deps is enough)
- No assignee/owner (single-player workflow)
- No tags/labels (use grep)
- No separate `status` file, `plan.md`, `log.md` per issue (one file is simpler)
- No JSON/YAML status files (frontmatter is enough)
