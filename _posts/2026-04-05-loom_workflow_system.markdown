### What Makes Loom Different from 1–3

Parley, issues, and vision are each **domain-specific** — they have a fixed schema and a known set of state transitions. Loom is the **generalization**: it's a workflow engine where the agent defines the schema, the transitions, and the triggers based on the task at hand.

This is the leap from "tool" to "substrate." And it's where things get both powerful and dangerous.

### The Core Architectural Question

The critical design tension is: **how much structure does loom impose vs. leave to the agent?**

Too much structure → you're reimplementing Jira/Asana in markdown, and the agent is just a fancy CLI.
Too little structure → the agent creates bespoke state formats every time, and humans can't navigate without asking the agent to explain its own files.

I think the sweet spot is something like:

```
loom/
  loom.yaml              # registry of active "threads" (projects/workflows)
  threads/
    quarterly-planning/
      thread.yaml         # metadata: status, created, triggers, due
      state.md            # human-readable current state
      log.md              # append-only reasoning log
      artifacts/          # any outputs (reports, drafts, etc.)
    hiring-pipeline/
      thread.yaml
      state.md
      log.md
      artifacts/
```

Where `thread.yaml` has a minimal common schema:

```yaml
id: quarterly-planning
status: active          # active | paused | completed | blocked
created: 2026-04-01
last_touched: 2026-04-05
trigger:
  schedule: "weekly/monday"   # or: on_change, manual, event
  condition: "when vision/roadmap.yaml changes"
owner: human            # or: agent, collaborative
summary: "Synthesize team updates into quarterly plan"
depends_on: []
```

And `state.md` is **freeform but agent-maintained** — this is where the agent externalizes its "mental state" about the workflow. The key insight: the agent writes state.md for its future self (and for humans), the same way a good engineer writes design docs.

### Why This Actually Works (and Why Now)

Three things converge:

1. **Coding agents are file-native.** They already think in terms of reading files, reasoning, writing files. You're not fighting the grain — you're aligning the workflow engine with the agent's natural modality.

2. **Git gives you durability + auditability for free.** Every state transition is a commit. You can revert. You can branch. You can diff. No database needed.

3. **Markdown is the universal interface.** Humans read it, agents read it, GitHub renders it, grep finds it. It's the lowest-common-denominator that actually works.

### The "Operating System" Analogy

I think the analogy is apt but worth being precise about. What you're building isn't an OS in the process-scheduling sense. It's more like a **file-system-based workflow kernel**, where:

- **Processes** = threads (active workflows)
- **System calls** = agent invocations (read state, reason, write state)
- **Scheduler** = trigger system (cron-like, event-driven, manual)
- **IPC** = file-based (one thread reads another thread's state.md)
- **Shell** = the human, who can `cat`, edit, or kick off any thread

The missing piece — and the hardest piece — is the **scheduler/trigger layer**. Right now you're describing manual invocation: human says "hey Claude, go work on the quarterly-planning thread." The real power comes when threads can:

- Self-schedule ("check back on this Monday")
- React to events ("when a new issue is created in issues/, triage it")
- Chain ("when hiring-pipeline reaches 'offer-sent', trigger onboarding thread")

This is where you'd need a lightweight daemon or cron job that scans `loom/threads/*/thread.yaml` for due triggers and invokes the agent. That's the "init system" of your OS.

### Honest Concerns

**State corruption.** When the agent rewrites `state.md`, it can lose information, hallucinate progress, or drift from reality. Mitigation: the append-only `log.md` creates an audit trail, and humans periodically review. But this requires discipline.

**Context window limits.** A long-running thread accumulates context. Eventually the agent can't hold the full history. You'll need a summarization/compaction strategy — maybe the agent periodically rewrites `state.md` as a condensed "current understanding" while `log.md` retains the full history (but is only consulted when needed).

**Agent reliability for scheduling.** Agents are good at reasoning but unreliable as schedulers. The trigger/cron layer should probably be a simple script, not agent-driven. Let dumb code handle "when," let the agent handle "what."

### What I'd Build First

If I were you, I'd start with:

1. **The thread schema** (`thread.yaml` + `state.md` + `log.md`) — nail the minimal structure
2. **A `loom` CLI or set of scripts** — `loom new <thread>`, `loom list`, `loom run <thread>` (which invokes Claude with the thread's context)
3. **One real workflow** — something you actually do weekly, like "synthesize team standup notes into a status update." Prove the pattern works before generalizing.
4. **The trigger layer** — a cron job that scans for due threads and invokes `loom run`

Skip building a UI. The repo *is* the UI. That's the whole point.

### The Bigger Picture

What you're describing — parley + issues + vision + loom — is essentially **externalizing the startup's cognitive processes into version-controlled, agent-manipulable files.** The startup's "brain" becomes inspectable, forkable, and collaborative between humans and agents.

This is philosophically closer to a **Zettelkasten for organizations** than to project management software. And the fact that it's built on git means it inherits all of git's collaboration primitives (branches, PRs, merges) for free.

The real question isn't whether this pattern works — I think it clearly does for a certain class of knowledge worker. The question is whether it **scales beyond the founder/power-user** who set it up. Can a new engineer join and navigate `loom/threads/` without a 30-minute walkthrough? That's the test.
